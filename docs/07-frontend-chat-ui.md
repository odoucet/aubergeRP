# 07 — Frontend Chat UI

## 1. Overview

The Chat UI is the primary user-facing interface. It provides:

- Character selection from the library.
- Conversation management (create, switch, delete).
- Real-time chat with streamed assistant responses.
- Inline display of images generated by the LLM (via markers — see [05 § 7](05-chat-and-conversations.md)).
- A clean, responsive layout.

MVP is single-user. Multi-user is out of scope (see [POST-MVP.md](POST-MVP.md)).

## 2. Technology

- Pure HTML + vanilla JavaScript (no framework, no build step).
- Plain CSS, responsive with media queries.
- Vendored libraries only — no CDN loads at runtime.
- `marked.js` (vendored) for Markdown rendering in messages.
- `fetch` for REST calls. SSE is read off a POST body via `fetch` + `ReadableStream` (native `EventSource` does not support POST or custom headers).

## 4. Components

### 4.1 Header Bar

- Application name/logo on the left.
- Link to Admin UI on the right.

### 4.2 Sidebar

Two togglable sections:

**Characters** — list of available characters (avatar + name). Click to select.

**Conversations** — list of conversations for the selected character (title + message count). "New Conversation" at the top. Click to switch. Delete button (with confirmation) on each.

### 4.3 Chat Area

- Character avatar + name in a small header.
- Scrollable message list, auto-scrolls to bottom on new messages.
- User messages aligned right; assistant messages left.
- Messages rendered as Markdown (`marked.js`).
- Images displayed inline within assistant messages (one or more per message).
- Typing indicator (animated dots) while the assistant is streaming.
- Image placeholder + spinner shown while an image is being generated.

### 4.4 Input Area

- Multi-line textarea, auto-expanding.
- `Send` button.
- Enter sends; Shift+Enter inserts a newline.
- Disabled while the assistant is streaming.

No manual "Generate Image" button — the LLM decides when to emit an image marker. See [05 § 7](05-chat-and-conversations.md).

### 4.5 Status Bar

- Connection indicator per connector type (text, image, video, audio).
  - Green check: connected (last known good).
  - Red X: configured but disconnected.
  - Dash: not configured.
- Polls `GET /api/health` every 30 seconds, **paused when the tab is hidden** (Page Visibility API).

## 5. User Flows

### 5.1 First-Time User

- Opens `http://localhost:8000`.
- Sidebar shows no characters; chat area shows: "No characters available. Import characters in the Admin panel." with a prominent link to `/admin/`.
- Status bar reflects connector state.

### 5.2 Starting a Conversation

- Click a character → if no conversations exist for that character, a new one is created automatically and the character's `first_mes` appears as the first message.

### 5.3 Sending a Message

1. User types, hits Enter (or Send).
2. The user message appears immediately (optimistic UI).
3. A typing indicator appears for the assistant.
4. The POST/SSE stream is opened via `fetch` + `ReadableStream`.
5. `token` events append text to the assistant bubble live.
6. `image_start` events show a placeholder + spinner inline in the assistant bubble.
7. `image_complete` events replace the placeholder with the image (`<img src="/api/images/…">`).
8. `image_failed` events replace the placeholder with a small inline error.
9. `done` event finalizes the message and removes the typing indicator.
10. `error` event shows an inline error below the last message.

### 5.4 Image Generation

Triggered by the LLM. The user simply asks conversationally ("show me the forest", "send a picture"). The backend handles marker detection and calls the image connector. See [05 § 7](05-chat-and-conversations.md).

The frontend responds to `image_start` / `image_complete` / `image_failed` SSE events in real time — no polling endpoint.

### 5.5 Switching Conversations

Click a conversation → load its messages (images included) into the chat area.

## 6. SSE Streaming in JavaScript

Because the message is sent via POST, we use `fetch` + `ReadableStream` rather than `EventSource`:

```javascript
const res = await fetch(`/api/chat/${conversationId}/message`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ content: message })
});

const reader = res.body.getReader();
const decoder = new TextDecoder();
// Parse SSE frames: "event: <name>\n" and "data: <json>\n\n".
// Dispatch to handlers: token, image_start, image_complete, image_failed, done, error.
```

A small SSE parser lives in `frontend/js/chat.js`.

## 7. Responsive Design

| Viewport | Behavior |
|---|---|
| ≥ 1024px | Sidebar always visible, 280px wide |
| 768–1023px | Sidebar collapsible via hamburger menu |
| < 768px | Sidebar is a drawer; input area fixed to bottom |

## 8. Accessibility

- Semantic HTML (`<main>`, `<nav>`, `<article>`, `<section>`).
- ARIA labels for interactive elements.
- Keyboard navigation (Tab, Enter to send).
- Color contrast meets WCAG AA.
- `alt` text on avatars and generated images.

## 9. Error States

| State | Display |
|---|---|
| No characters | Empty-state message + link to `/admin/` |
| Text connector not configured / disconnected | Red status; send shows inline error |
| Image connector not configured | Dash status; `image_failed` SSE shown inline when markers are emitted |
| Image generation failed | Small inline error where the image would appear |
| LLM error during streaming | Inline error below the last message; retry button |
| Network error | Toast/banner at the top |
